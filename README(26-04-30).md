## 2. 核心改动详解

### 2.1 从两段式到单进程闭环

| 阶段 | 之前 | 现在 |
|------|------|------|
| 架构 | Habitat 生成图片 → 文件传输 → VLM 推理 | 同一进程内循环：`Habitat → VLM → Habitat → VLM → ...` |
| 脚本 | `habitat-vlm/loop-habitat.py` + `vlm_nav.py`（双环境） | `habitat-vlm/loop_navigation.py`（单环境） |
| 通信 | 文件系统（JPEG 写盘 → 读盘） | 内存 PIL Image 直接传递 |
| 效率 | 每次启动需加载 7B 模型 ~5.5GB | 模型一次性加载，后续推理 ~1.1s/步 |

单进程闭环的关键依赖：
- 在 `habitat` conda 环境（Python 3.9）中额外安装 transformers + torch + qwen-vl-utils + bitsandbytes
- 使用 `max_memory={0: 6_000_000_000}` 显存上限控制，避免 Habitat 渲染和 VLM 推理争抢显存

### 2.2 模型切换：7B → 2B

| 指标 | Qwen2-VL-7B (4-bit) | Qwen2-VL-2B (4-bit) |
|------|---------------------|---------------------|
| 显存占用 | ~5.5 GB | ~1.5 GB |
| 推理速度 | ~2.5s/step | ~1.1-1.25s/step |
| 动作解析 | 偶有拼写错误 | 同样稳定，输出更简洁 |

**结论**：在导航任务中，2B 模型的表现与 7B 无明显差距，且显存占用更少，为 Habitat 渲染留出充足空间。实际从 7B 切换到 2B 后，未出现因模型容量下降导致的导航失能。

### 2.3 工程化改进

#### 场景配置系统

```python
SCENES = {
    "apartment": {
        "path": ".../apartment_1.glb",
        "task": "找到通往卧室的门",
        "max_steps": 30,
    },
    "vangogh": {
        "path": ".../van-gogh-room.glb",
        "task": "探索整个房间",
        "max_steps": 30,
    },
    "castle": {
        "path": ".../skokloster-castle.glb",
        "task": "找到城堡的大门",
        "max_steps": 50,
    },
}
SCENE_NAME = "castle"  # 改这一行即可切换场景
```

- 每个场景独立配置：路径、任务描述、最大步数
- Prompt 自动适配场景名（"公寓"/"梵高房间"/"城堡"）

#### 输出隔离

```
habitat-vlm/
├── output_apartment/   # frame_000.jpg ~ frame_030.jpg + navigation_report.txt
├── output_vangogh/     # frame_000.jpg ~ frame_030.jpg + navigation_report.txt
├── output_castle/      # frame_000.jpg ~ frame_050.jpg + navigation_report.txt
└── loop_navigation.py
```

每次运行自主创建 `output_{场景名}/` 目录，防止不同实验的数据互相覆盖。

#### 撞墙检测

```python
consecutive_forward = 0
if action == "move_forward":
    consecutive_forward += 1
if consecutive_forward >= 3:  # 连续3次前进无位移 → 强制转向
    action = "turn_left"
```

防止 VLM 在障碍物前反复输出 move_forward 导致死锁。

### 2.4 图片预处理优化

| 参数 | 之前 | 现在 |
|------|------|------|
| 最大分辨率 | 560px | 784px |
| 对齐 | 28 的倍数 | 28 的倍数 |
| 图片质量 | JPEG 默认 (75) | JPEG quality=95 |

784px 在室内场景中能保留更多远处走廊/门的细节，quality=95 减少 JPEG 压缩对 VLM 识别的干扰。

### 2.5 Prompt 工程迭代

历史上下文从 3 步扩展到 6 步：

```
第 12 步，之前动作：move_forward -> move_forward -> turn_left -> turn_left -> move_forward -> move_forward

观察画面，分析周围环境结构，然后选择下一步动作：
- 如果前方看起来是走廊或开放空间 → move_forward
- 如果前方是墙壁，左侧有空间 → turn_left
- 如果前方是墙壁，右侧有空间 → turn_right

回答格式：只输出一个动作名称。
```

6 步历史提供了更充分的动作模式上下文，减少 VLM 的短期振荡。

## 3. 三次完整跑测结果

### 3.1 公寓（30步）

| 项目 | 值 |
|------|-----|
| 初始位置 | (1.2, 0.18, 1.5) |
| 探索区域 | 走廊 + 客厅区域 |
| VLM 提前 stop | 否（满30步） |
| 典型行为 | 沿走廊推进，撞墙后转向 |

### 3.2 梵高房间（30步）

| 项目 | 值 |
|------|-----|
| 初始位置 | (3.10, 0.18, 0.97) |
| 最终位置 | (1.13, 0.18, 1.63) |
| 推理速度 | ~1.03s/步 |
| 特殊现象 | 在 (-7.8, 5.6) 附近卡了约 13 步后脱困 |

梵高房间是单间，空间较小。agent 在房间中部多次转向后找到新方向，成功穿越房间。

### 3.3 城堡（50步）

| 项目 | 值 |
|------|-----|
| 初始位置 | (-1.37, 0.08, 8.61) |
| 最终位置 | (0.92, 0.07, 8.09) |
| 推理速度 | ~1.24s/步 |
| 轨迹形态 | **完整环线** |

轨迹的三个阶段：
1. **Phase 1**（Step 0–17）：从入口向西南角推进
2. **Phase 2**（Step 18–30）：在西南角卡住，连续转向+微移
3. **Phase 3**（Step 32–50）：成功脱困，沿北向折返到初始位置对面

50 步覆盖了城堡大半个内部区域，2B 模型全程稳定输出，未出现任何异常退出。

## 4. 新发现与遗留问题

### 4.1 新发现

- **2B 模型足够**：在纯 RGB 室内导航任务中，2B 的表现与 7B 无明显差异，推理更快、显存更省
- **撞墙检测有效**：捕获了多次卡墙场景，强制转向使 agent 能脱困
- **高分辨率有帮助**：从 560→784 之后，VLM 对远处走廊的判断更清晰
- **城堡大场景可探索**：50 步覆盖半个城堡，agent 能走通环路

### 4.2 遗留问题

1. **局部振荡**：在开阔区域 VLM 仍会连续转向，造成步数浪费（如 step 7~14 连续转向）
2. **无空间记忆**：agent 不知道自己走过哪里，反复经过同一位置
3. **纯 RGB 缺深度**：无法区分"前方是走廊"和"前方是玻璃门"，该停不停或该走不走
4. **城堡卡角**：西南角卡了 13 步才脱困，撞墙检测还不够灵敏
